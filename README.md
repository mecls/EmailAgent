# EmailAgent

A read-only, multi-tenant Gmail assistant. Clients sign in with Google, grant
`gmail.readonly`, a background worker indexes the last year of mail, and an
LLM agent answers questions about the inbox, ships a daily **morning brief**, and
takes **voice** input. It never sends or modifies mail — read-only scope and the
absence of any action tools are the core safety property, by design.

This README doubles as a tour of how the system is built and the decisions behind
it.

## What it does

- **Chat with your inbox** — a streaming, tool-calling agent answers "who's
  waiting on me?", "catch me up on \<person\>", "anything urgent I missed?". It
  reasons over three read-only tools (below) and cites the subject + sender so
  answers are verifiable.
- **Morning brief** — a scheduled, per-account digest of what matters (leads,
  clients, ops, things to sign), generated by the LLM and delivered by email at a
  time the user picks; runnable on demand.
- **Voice dictation (on-device)** — a Wispr Flow–style push-to-talk: tap the mic,
  speak, and the question is transcribed **locally in the browser** and dropped
  into the composer to review and send. See [Voice](#voice-on-device-dictation).
- **Read-only by construction** — there are no send/modify/label tools to call,
  the OAuth scope is `gmail.readonly`, and every answer is framed as information,
  never an action.

## Stack

Next.js 16 (App Router, React 19, TS strict, Tailwind v4) · installable **PWA**,
mobile-first · Supabase (Postgres + Auth + pgvector + Vault + RLS) · Inngest
(durable jobs + crons) · OpenAI-compatible LLM (Ollama Cloud, via
`LLM_BASE_URL`/`LLM_API_KEY`/`LLM_MODEL` — model must support tool calling) for
the agent loop + brief · Supabase `embed` Edge Function (gte-small, 384-dim
embeddings) · `@huggingface/transformers` (in-browser Whisper for voice) · Resend
(brief email).

## Architecture (short)

- **Client** is thin: Google sign-in, Gmail connect, a streaming chat box, a brief
  feed. Reads tenant data through Supabase RLS (user JWT). Calls a server-side
  agent route for chat (SSE).
- **Supabase** is the source of truth: Auth, Postgres (RLS per `account_id`),
  pgvector for semantic search, and Vault for the encrypted Gmail refresh token.
- **Inngest** is the worker + scheduler: `index.kickoff → page → batch →
  derive-status`, plus `freshness.poll`, `brief.scheduler → brief.generate`, and
  token refresh. Every job is account-scoped and uses the service-role key, so it
  filters by `account_id` itself (RLS is bypassed deliberately, never by accident).
- **Index-light**: store metadata + snippet + embeddings; `body_text` stays NULL
  and is re-fetched from Gmail on demand — the smallest possible breach blast
  radius.

### The agent loop

A multi-turn, streaming loop over the OpenAI-compatible endpoint with three
**read-only** tools, all hard-scoped to a server-derived `account_id` the model
can never override:

- `search_email` — semantic search over recent, indexed mail (pgvector).
- `search_gmail_live` — live keyword search over full Gmail history, for older
  mail the index doesn't cover.
- `query_email` — structured filters (thread reply-state, direction, sender, date)
  for "who owes a reply", overdue threads, etc.

Tool output is treated as **untrusted data**: the system prompt explicitly refuses
to follow instructions found inside email bodies (prompt-injection defense), and
the user-facing copy is kept jargon-free.

## Voice (on-device dictation)

Voice input runs the open-source **Whisper** model **entirely in the browser** via
`transformers.js`, inside a Web Worker:

- **Free and private** — no API key, no per-use cost, and the **audio never leaves
  the device**. Only the model weights download once (from a public CDN) and are
  cached; transcription then works even offline.
- **How it works** — `getUserMedia` → `MediaRecorder` → decode and resample to
  16 kHz mono → the worker transcribes off the main thread → the text fills the
  composer for review/edit. Reliable inside an installed iOS PWA, where the
  browser's native speech API is not.
- **Why this trade-off** — a one-time model download and a few seconds of
  on-device compute, chosen over a paid cloud transcriber, to keep the product
  free and the user's audio fully private. The model id is a single constant, so
  swapping to a larger/multilingual model — or a cloud transcriber for speed — is
  a one-line change.

## How it's built (engineering notes)

- **Security and tenancy first.** Account id is always re-derived from the verified
  session, never taken from the client; the browser uses RLS, the worker uses the
  service-role key and self-filters. Tokens live in Vault, not the app DB.
- **Durable, idempotent background work.** Indexing and briefs are Inngest jobs
  with crons, retries, and account-scoped fan-out — safe to re-run.
- **Minimize blast radius.** Read-only scope, no action tools, index-light storage,
  on-device voice. Each is a deliberate "least data / least power" choice.
- **Pragmatic, cost-aware.** Ollama Cloud for the agent and on-device Whisper for
  voice keep recurring cost near zero without sacrificing the core UX.
- **Honest UX.** Streaming answers with citations, no internal jargon, and empty/
  error states that explain themselves.
